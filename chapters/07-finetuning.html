🎯 Fine-Tuning LLMs

Pre-trained LLMs are powerful generalists. Fine-tuning makes them specialists — teaching GPT-style models to follow instructions, adopt a persona, or master a domain. This chapter covers every major approach, from full fine-tuning to efficient adapters.

Why Fine-Tune?

Pre-training vs Fine-tuning: A base LLM like LLaMA 3 is trained to predict the next token in raw text. It's not trained to be helpful, to follow instructions, or to refuse harmful requests. Fine-tuning shapes behavior without re-training from scratch.

Common reasons to fine-tune:

  • Instruction following — teach the model to respond to commands
  • Domain adaptation — medical, legal, coding, customer support
  • Style/tone — make a model match your brand voice
  • Format alignment — always output JSON, always cite sources
  • Safety — reduce harmful outputs for your deployment context

Approach 1: Full Fine-Tuning

Update all model parameters on your dataset. Most powerful, but expensive:

  • A 7B model needs ~56GB of VRAM just for weights in fp16
  • With optimizer states (Adam), multiply by 4–8×
  • Requires A100/H100 class hardware for any large model
  • Risk of catastrophic forgetting — overwriting general knowledge
# Full fine-tuning with Hugging Face Trainer
from transformers import AutoModelForCausalLM, AutoTokenizer, Trainer, TrainingArguments
from datasets import load_dataset

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.2-1B")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-1B")

dataset = load_dataset("your-dataset")

def tokenize(examples):
    return tokenizer(examples["text"], truncation=True, max_length=512)

tokenized = dataset.map(tokenize, batched=True)

args = TrainingArguments(
    output_dir="./output",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=8,   # simulate larger batch
    num_train_epochs=3,
    learning_rate=2e-5,
    fp16=True,                        # half precision to save VRAM
    save_strategy="epoch",
)

trainer = Trainer(model=model, args=args, train_dataset=tokenized["train"])
trainer.train()

Approach 2: LoRA — Low-Rank Adaptation

Key insight: Instead of modifying all weights, LoRA freezes the original model and injects small trainable "adapter" matrices into each layer. The adapters have far fewer parameters — typically 1–10% of the full model size — so training is fast and cheap.

How LoRA works mathematically

Each weight matrix W (frozen) gets a parallel low-rank update:

W' = W + ΔW = W + (A × B)

Where:
  W  ∈ R^(d × d)   — original frozen weights
  A  ∈ R^(d × r)   — trainable (random init)
  B  ∈ R^(r × d)   — trainable (zero init, so ΔW starts at 0)
  r  = rank         — typically 4, 8, 16, or 64

The rank r is the key hyperparameter. Higher rank = more capacity = more VRAM.

# LoRA fine-tuning with PEFT + bitsandbytes (4-bit quantization)
# pip install peft bitsandbytes transformers accelerate datasets

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model, TaskType
import torch

# Load model in 4-bit (QLoRA) — runs on consumer GPU
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_use_double_quant=True,
    bnb_4bit_compute_dtype=torch.bfloat16,
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.2-3B-Instruct",
    quantization_config=bnb_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-3B-Instruct")

# Attach LoRA adapters
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                     # rank
    lora_alpha=32,             # scaling factor
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],  # which layers to adapt
    bias="none",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 4,194,304 || all params: 3,213,623,296 || trainable%: 0.13%

QLoRA: Quantize + LoRA

QLoRA (Dettmers et al., 2023) combines 4-bit quantization with LoRA. A 65B model that normally needs 8× A100s can be fine-tuned on a single 48GB GPU. This democratized LLM fine-tuning enormously.

Approach 3: Instruction Tuning

Instruction tuning teaches a model to follow natural language commands. The key is data format — you need prompt/response pairs:

# Alpaca-style instruction format
example = {
    "instruction": "Translate the following to French.",
    "input": "The weather is nice today.",
    "output": "Le temps est agréable aujourd'hui."
}

# Format into a single training string
def format_prompt(example):
    return f"""### Instruction:
{example['instruction']}

### Input:
{example['input']}

### Response:
{example['output']}</s>"""

# Modern chat format (preferred for chat models)
messages = [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "What is the capital of France?"},
    {"role": "assistant", "content": "The capital of France is Paris."},
]
# Apply the tokenizer's chat template
formatted = tokenizer.apply_chat_template(messages, tokenize=False)

Good instruction datasets include: Alpaca, FLAN, Open Hermes, ShareGPT, and Dolly.

Approach 4: RLHF (Reinforcement Learning from Human Feedback)

This is how ChatGPT, Claude, and Gemini are aligned. RLHF takes a supervised fine-tuned model and further trains it to produce outputs that humans prefer — using human ratings as a training signal.

The three-stage pipeline:

1
Supervised Fine-Tuning (SFT)

Fine-tune the base model on high-quality prompt/response demonstrations. This gives the model a starting point for human-preferred behavior.

2
Reward Model Training

Humans compare pairs of responses and pick the better one. A separate "reward model" is trained to predict these preferences — outputting a scalar score for any (prompt, response) pair.

3
PPO / RL Optimization

The SFT model is further trained via Proximal Policy Optimization (PPO) to maximize the reward model's score — while a KL divergence penalty prevents it from drifting too far from the SFT model.

DPO: A Simpler Alternative

Direct Preference Optimization (DPO, 2023) skips the reward model entirely. It directly trains the LLM on preference pairs using a re-derived loss function that implicitly encodes the same objective as RLHF, but with standard supervised training:

from trl import DPOTrainer, DPOConfig

# Dataset format: each row has prompt, chosen response, rejected response
# {
#   "prompt": "Explain quantum computing.",
#   "chosen": "Quantum computing uses qubits...",  ← preferred
#   "rejected": "Quantum computing is complicated." ← not preferred
# }

dpo_config = DPOConfig(
    beta=0.1,      # KL penalty strength
    output_dir="./dpo-output",
    per_device_train_batch_size=2,
    max_length=512,
)

trainer = DPOTrainer(
    model=model,
    args=dpo_config,
    train_dataset=preference_dataset,
    tokenizer=tokenizer,
)
trainer.train()
