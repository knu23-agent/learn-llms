🤖 Building Agentic AI

An LLM that just answers questions is a chatbot. An LLM that can take actions, use tools, remember state, and pursue goals across multiple steps — that's an agent. This chapter explains how tools like Claude Code, Devin, and Cursor work, and how to build your own.

What Makes Something an "Agent"?

An agent is an LLM in a loop. It perceives its environment (context), decides what to do (tool call or final answer), acts (executes the tool), observes the result, and loops until the task is complete.

The key properties that distinguish agents from plain LLMs:

  • Tool use — can call external functions (run code, search the web, read files)
  • Multi-step reasoning — plans and executes sequences of actions
  • State / memory — remembers what happened earlier in the task
  • Self-correction — sees tool output and adjusts plans accordingly
  • Goal-directedness — works toward a specified objective, not just next-token prediction

The ReAct Pattern

ReAct (Reasoning + Acting) is the most common agent architecture. The model alternates between thinking and doing:

Thought: I need to find the population of Tokyo.
Action: search("Tokyo population 2024")
Observation: Tokyo has a population of approximately 13.96 million (city).

Thought: I have the answer. I should also note the greater metro area.
Action: search("Tokyo metropolitan area population")
Observation: Greater Tokyo Area has ~37 million people.

Thought: Now I can answer the question.
Final Answer: Tokyo city has ~14M people; the greater metro area ~37M.

The model generates "Thought:" and "Action:" tokens; the framework parses the action, runs it, and appends "Observation:" with the result — feeding everything back into the context for the next iteration.

Tool Calling in Practice

Modern LLM APIs support structured tool calling natively. You define tools as JSON schemas; the model decides when and how to call them:

from openai import OpenAI
import json

client = OpenAI()

# Define tools as JSON schemas
tools = [
    {
        "type": "function",
        "function": {
            "name": "run_python",
            "description": "Execute Python code and return stdout/stderr",
            "parameters": {
                "type": "object",
                "properties": {
                    "code": {"type": "string", "description": "Python code to execute"}
                },
                "required": ["code"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "read_file",
            "description": "Read a file from the filesystem",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {"type": "string", "description": "File path to read"}
                },
                "required": ["path"]
            }
        }
    }
]

The Agent Loop

The core of any agent is a loop: call the LLM, handle tool calls, feed results back:

import subprocess, json
from openai import OpenAI

client = OpenAI()

# Tool implementations
def run_python(code):
    result = subprocess.run(
        ["python3", "-c", code],
        capture_output=True, text=True, timeout=10
    )
    return result.stdout + result.stderr

def read_file(path):
    try:
        with open(path) as f: return f.read()
    except Exception as e: return str(e)

TOOL_MAP = {"run_python": run_python, "read_file": read_file}

def run_agent(user_task, max_steps=10):
    messages = [
        {"role": "system", "content": "You are a coding agent. Use tools to complete tasks."},
        {"role": "user", "content": user_task},
    ]

    for step in range(max_steps):
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
            tool_choice="auto",
        )
        msg = response.choices[0].message
        messages.append(msg)

        # No tool calls → final answer
        if not msg.tool_calls:
            return msg.content

        # Execute each tool call
        for call in msg.tool_calls:
            fn = call.function.name
            args = json.loads(call.function.arguments)
            result = TOOL_MAP[fn](**args)   # call the real function

            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result,
            })

    return "Max steps reached."

# Run it!
answer = run_agent("Write a Python function to calculate Fibonacci numbers, then test it.")
print(answer)

How Claude Code Works (Conceptually)

Claude Code, Cursor, Devin, and similar tools are all variants of this same loop — but with more sophisticated tools, memory systems, and safety layers.

What a coding agent like Claude Code does internally:

1
Context loading

Reads the user's task + loads relevant files, git history, directory structure. Builds a rich context window: "here is the codebase, here is what you need to do."

2
Planning

Reasons about what steps are needed. May produce a plan explicitly ("first I'll read X, then edit Y, then run tests") or implicitly through tool selection.

3
Tool execution

Calls tools: read_file, write_file, run_bash, search_codebase. Each result is fed back into context.

4
Verification

Runs tests, checks output, reads compiler errors. If something fails, reasons about why and tries a different approach.

5
Completion / escalation

When confident the task is done, returns the result. If stuck or uncertain, asks the human for clarification.

Memory Systems

Agents need different types of memory to work on complex tasks:

TypeWhat it isHow to implement
In-context The conversation history in the prompt Just keep appending to messages
External (files) Notes written to disk between sessions write_file("memory.md", notes)
Vector store Semantic search over past events/docs Embed text → store in Chroma/Pinecone → retrieve by similarity
Episodic Structured log of past actions and outcomes Append JSON records, summarize periodically

Building a Minimal Coding Agent: Full Example

Here's a complete, runnable coding agent using the Anthropic API and Claude:

# pip install anthropic
import anthropic, subprocess, os, json

client = anthropic.Anthropic()

# --- Tool definitions ---
TOOLS = [
    {
        "name": "bash",
        "description": "Run a bash command. Returns stdout + stderr.",
        "input_schema": {
            "type": "object",
            "properties": {"command": {"type": "string"}},
            "required": ["command"]
        }
    },
    {
        "name": "write_file",
        "description": "Write content to a file.",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["path", "content"]
        }
    },
    {
        "name": "read_file",
        "description": "Read a file's contents.",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"]
        }
    }
]

# --- Tool implementations ---
def run_bash(command):
    try:
        r = subprocess.run(command, shell=True, capture_output=True,
                           text=True, timeout=30)
        out = r.stdout + r.stderr
        return out[:4000] if out else "(no output)"
    except subprocess.TimeoutExpired:
        return "Error: command timed out"

def write_file(path, content):
    os.makedirs(os.path.dirname(path) or ".", exist_ok=True)
    with open(path, "w") as f: f.write(content)
    return f"Written {len(content)} chars to {path}"

def read_file(path):
    try:
        with open(path) as f: return f.read()[:8000]
    except FileNotFoundError:
        return f"File not found: {path}"

TOOL_MAP = {"bash": run_bash, "write_file": write_file, "read_file": read_file}

# --- Agent loop ---
def run_coding_agent(task):
    messages = [{"role": "user", "content": task}]

    for step in range(20):
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            system="""You are an expert coding agent. 
Think step-by-step. Use tools to write code, run it, and fix errors.
When the task is complete, summarize what you did.""",
            tools=TOOLS,
            messages=messages,
        )

        # Add assistant turn to history
        messages.append({"role": "assistant", "content": response.content})

        # Stop if done
        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, "text"):
                    print(block.text)
            return

        # Process tool calls
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                print(f"[{block.name}] {json.dumps(block.input)[:80]}...")
                result = TOOL_MAP[block.name](**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })

        messages.append({"role": "user", "content": tool_results})

# Try it!
run_coding_agent("""
Create a Python script that:
1. Generates a list of 10 random numbers between 1 and 100
2. Sorts them
3. Computes mean and median
4. Saves the results to results.txt
5. Run it and show the output.
""")

Safety and Control

Agents are powerful and risky. A bug or misunderstanding can lead to deleting files, sending emails, or making API calls you didn't intend. Always build in human oversight, especially for irreversible actions.

Essential safety patterns:

  • Sandboxing — run code in Docker containers, not on your host machine
  • Confirmation for destructive actions — ask before deleting, deploying, or sending
  • Budget limits — cap API calls, tokens, and time per task
  • Audit logging — record every action the agent takes
  • Scoped permissions — give the agent only the access it needs for the task
  • Reversibility — prefer operations that can be undone (git commits, not rm -rf)
# Sandboxed execution with Docker
def run_python_sandboxed(code):
    result = subprocess.run([
        "docker", "run", "--rm",
        "--network=none",          # no internet access
        "--memory=256m",            # memory limit
        "--cpus=1",                  # CPU limit
        "--read-only",              # read-only filesystem
        "python:3.11-slim",
        "python3", "-c", code
    ], capture_output=True, text=True, timeout=15)
    return result.stdout + result.stderr
